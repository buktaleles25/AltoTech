# กรณีศึกษา: Trust Boundary ของระบบ Save Editing และ Pokémon HOME

> เอกสารนี้เป็น **กรณีศึกษาเชิงสถาปัตยกรรมและความปลอดภัย** ประกอบกับเครื่องมือ
> `pokemon-batch-injector` ในรีโพนี้ จุดประสงค์คือใช้ระบบของซีรีส์ Pokémon เป็น
> ตัวอย่างในการอธิบายหลักการ **"อย่าเชื่อ client" (never trust the client)** และ
> การวาง trust boundary ของระบบ client–server
>
> เอกสารนี้ **ไม่ใช่** คู่มือการหลบเลี่ยงระบบตรวจจับหรือการแบนของบริการใดๆ และ
> ไม่สนับสนุนการละเมิดข้อตกลงการใช้งาน (ToS) ของผู้ให้บริการ

---

## 1. ขอบเขตและคำถามตั้งต้น

ผู้เริ่มต้นแก้ไขเซฟเกม (save editing) มักตั้งคำถามว่า:

> "ในเมื่อเราแก้ค่าในไฟล์เซฟได้ทุกอย่าง ทำไมเรา 'ยัด' โปเกมอนเข้า Pokémon HOME
> ตรงๆ ไม่ได้?"

คำตอบไม่ได้อยู่ที่ "ยังหาวิธีไม่เจอ" แต่อยู่ที่ **การออกแบบขอบเขตความเชื่อใจ
(trust boundary)** ของระบบ ซึ่งเป็นหลักการความปลอดภัยพื้นฐานที่ใช้ได้กับทุกระบบ
client–server ไม่ใช่แค่เกม

---

## 2. โครงสร้างระบบเป็นชั้น (Layered View)

```
┌─────────────────────────────────────────────────────────┐
│  Layer 3 — Pokémon HOME cloud   (server-authoritative)    │
│  • ข้อมูลตัวจริงอยู่บนเซิร์ฟเวอร์ผู้ให้บริการ                  │
│  • ตรวจสอบความถูกต้องฝั่ง server ก่อนรับเข้า                  │
└───────────────▲─────────────────────────────────────────┘
                │  ต่อผ่านเกมที่ "เชื่อถือได้" + เน็ตเวิร์กที่รับรอง
┌───────────────┴─────────────────────────────────────────┐
│  Layer 2 — Game on console      (semi-trusted client)     │
│  • รันบนฮาร์ดแวร์แท้ + เฟิร์มแวร์ที่ลงลายเซ็น                │
│  • มี key/certificate สำหรับคุยกับ Layer 3                  │
└───────────────▲─────────────────────────────────────────┘
                │  อ่าน/เขียนไฟล์เซฟ
┌───────────────┴─────────────────────────────────────────┐
│  Layer 1 — Save file            (UNtrusted / ผู้ใช้คุม 100%)│
│  • ไฟล์อยู่ในมือผู้ใช้ แก้ได้ทุกบิต (เช่นด้วย PKHeX)          │
└─────────────────────────────────────────────────────────┘
```

หัวใจของกรณีศึกษาคือ **เส้นแบ่งระหว่าง Layer 1 กับ Layer 3** และทำไมการควบคุม
Layer 1 ได้ 100% ถึง **ไม่** แปลว่าควบคุม Layer 3 ได้

---

## 3. Layer 1 — ทำไม "แก้เซฟ" ถึงทำได้ง่าย

ไฟล์เซฟคือข้อมูลที่อยู่ในเครื่องผู้ใช้ทั้งหมด ผู้ใช้จึง:

- อ่าน/เขียน/ถอดรหัสได้ทุกไบต์
- เครื่องมืออย่าง PKHeX เข้าใจ schema ของไฟล์ จึงแก้ค่าได้ตามต้องการ
- **ไม่มี trust ใดๆ ในชั้นนี้** — ระบบไม่เคยถือว่าไฟล์เซฟ "พูดความจริง"

เครื่องมือในรีโพนี้ (`pokemon-batch-injector`) ทำงานทั้งหมดอยู่ใน Layer 1:
รับ Showdown set → สร้าง `PKM` → เขียนลงช่องกล่องในไฟล์เซฟ นี่คือสาเหตุที่มัน
"ทำได้" — เพราะไม่มีใครต้องเชื่อผลลัพธ์ของมันนอกจากตัวผู้ใช้เอง

> **บทเรียน:** สิ่งที่อยู่ในมือ client แก้ได้เสมอ การออกแบบระบบจึงต้องสมมติว่า
> ข้อมูลฝั่ง client "อาจถูกปลอม" ไว้ก่อน

---

## 4. Layer 3 — ทำไม "ยัดเข้า HOME ตรงๆ" ถึงทำไม่ได้

Pokémon HOME เป็น **cloud service ฝั่งเซิร์ฟเวอร์**:

- ตัวโปเกมอนของจริงถูกเก็บบนเซิร์ฟเวอร์ผู้ให้บริการ ไม่ใช่ไฟล์ในเครื่องเรา
- การสื่อสารถูกเข้ารหัสและต้องผ่านการพิสูจน์ตัวตน (authentication)
- เซิร์ฟเวอร์ทำ **validation ของตัวเอง** ก่อนจะรับข้อมูลเข้า ไม่เชื่อ client

เพราะ "แหล่งความจริง (source of truth)" อยู่ฝั่งเซิร์ฟเวอร์ การจะใส่ข้อมูลเข้าไป
จึงต้อง **ผ่านช่องทางที่เซิร์ฟเวอร์ยอมรับ** เท่านั้น ไม่มี "ไฟล์ HOME" ในเครื่อง
ให้เปิดแก้

> **บทเรียน:** เมื่อ source of truth อยู่ฝั่ง server การควบคุม client ได้เต็มที่
> ก็ไม่ทะลุ trust boundary นี้ ตราบใดที่ server ตรวจสอบข้อมูลขาเข้าเอง

---

## 5. Layer 2 — "ประตู" ที่ถูกควบคุมด้วยฮาร์ดแวร์ + ลายเซ็น

ข้อมูลจะข้ามจาก Layer 1 ขึ้น Layer 3 ได้ **ต้องผ่านเกมที่รันบน Layer 2** ซึ่งถูก
gate ด้วย:

- **ฮาร์ดแวร์แท้ + เฟิร์มแวร์ที่ลงลายเซ็น** — ผู้ให้บริการตรวจได้ว่าเครื่องและ
  ระบบยังไม่ถูกดัดแปลง
- **key / certificate** ที่ฝังในเครื่องแท้ ใช้พิสูจน์ตัวตนกับ Layer 3

นี่อธิบายตารางต่อไปนี้ (มุม "ทำไม" ไม่ใช่ "ทำยังไง"):

| สภาพแวดล้อม | ข้ามขึ้น HOME ได้ไหม | เหตุผลเชิงสถาปัตยกรรม |
|---|---|---|
| แอป HOME บนมือถือ | ❌ | เป็น *client* จัดการของที่อยู่ใน HOME แล้ว ไม่ใช่ช่อง ingress |
| Emulator บน PC | ❌ | ไม่มี key/certificate ที่ถูกต้อง → ถูกกันออกจากบริการออนไลน์ |
| Console ปกติ (ไม่ดัดแปลง) | ❌ ตั้งแต่ต้น | แก้เซฟใน Layer 1 ไม่ได้เพราะระบบไฟล์ถูกล็อก |
| Console ที่ดัดแปลง (CFW) | ⚠️ ในทางเทคนิคเป็นไปได้ | ทำลาย gate ของ Layer 2 — แต่ผิด ToS และเสี่ยงถูกแบน |

จุดสำคัญ: **emulator inject เซฟได้ แต่ขึ้น HOME ไม่ได้** เพราะมันอยู่ใน Layer 1
ได้ แต่ปลอมตัวเป็น Layer 2 ที่ "เชื่อถือได้" ไม่ได้ — นี่คือ trust boundary ทำงาน
ตามที่ออกแบบไว้พอดี

---

## 6. Defense in Depth: บทบาทของ "Legality Check"

แม้จะทะลุ Layer 2 มาได้ (กรณี CFW) ก็ยังเจอ **การตรวจสอบความถูกต้องฝั่งเซิร์ฟเวอร์**
อีกชั้น นี่คือหลักการ **defense in depth** — ป้องกันหลายชั้น ไม่พึ่งชั้นเดียว

เปรียบเทียบสองตัวตรวจ:

| | PKHeX `LegalityAnalysis` (Layer 1) | การตรวจฝั่ง HOME (Layer 3) |
|---|---|---|
| ทำงานที่ไหน | ในเครื่องผู้ใช้ | บนเซิร์ฟเวอร์ |
| ใครควบคุม | ผู้ใช้ | ผู้ให้บริการ |
| จุดประสงค์ | ช่วยผู้ใช้สร้างข้อมูลให้สอดคล้องกฎ | ตัดสินขั้นสุดท้ายว่าจะรับหรือไม่ |
| เชื่อถือได้แค่ไหนต่อระบบ | ไม่ถือเป็นข้อยุติ | เป็นข้อยุติ |

เครื่องมือในรีโพนี้ตั้งใจ inject **เฉพาะตัวที่ผ่าน `LegalityAnalysis` 100%**
ซึ่งเป็น *best practice* ฝั่ง client (สร้างข้อมูลให้ถูกต้องที่สุดเท่าที่รู้) —
แต่ **"ผ่าน PKHeX" ไม่เท่ากับ "ผ่าน server"** เพราะ:

- กฎฝั่งเซิร์ฟเวอร์อาจเข้มกว่า/อัปเดตบ่อยกว่า
- เซิร์ฟเวอร์มีข้อมูล/สัญญาณที่ client มองไม่เห็น
- ผู้ให้บริการเปลี่ยนกติกาได้ตลอด

> **บทเรียน:** การ validate ฝั่ง client มีไว้เพื่อ UX และลด error ไม่ใช่เพื่อความ
> ปลอดภัย การตัดสินที่มีผลจริงต้องทำฝั่ง server เสมอ

---

## 7. มิติด้านความเสี่ยงและจริยธรรม

- **ละเมิด ToS** ของผู้ให้บริการโดยตรง
- เสี่ยงถูก **ระงับบัญชี / ระงับการเข้าถึงบริการ / แบนเครื่อง**
- ถ้านำไป **แลกเปลี่ยนกับผู้เล่นอื่น** = ส่งผลกระทบต่อ ecosystem ที่คนอื่นใช้ร่วมกัน
  ไม่ใช่แค่ความเสี่ยงของตัวเอง

ด้วยเหตุนี้ การวิเคราะห์ในเอกสารนี้จึงหยุดที่ "ทำไมระบบถึงออกแบบมาแบบนี้" และ
**ไม่ลงรายละเอียดวิธีหลบการตรวจจับ** ซึ่งจะเป็นการช่วยละเมิดบริการที่ผู้อื่นใช้ร่วมกัน

---

## 8. บทเรียนเชิงวิศวกรรม (นำไปใช้ต่อได้กับระบบทั่วไป)

1. **Never trust the client.** อะไรที่อยู่ในมือผู้ใช้ ถือว่าแก้ได้หมด (ไฟล์เซฟ =
   ตัวอย่างชั้นดี)
2. **วาง source of truth ไว้ฝั่งที่คุณควบคุม.** ถ้าความจริงอยู่บน server การ
   ควบคุม client ของผู้โจมตีก็ไม่ทะลุขอบเขต
3. **Validate ที่ฝั่ง server เสมอ.** client-side validation มีไว้เพื่อ UX ไม่ใช่
   security
4. **ใช้ defense in depth.** ฮาร์ดแวร์/ลายเซ็น (authentication) + การตรวจข้อมูล
   (validation) หลายชั้น ดีกว่าพึ่งชั้นเดียว
5. **gate ด้วยการพิสูจน์ตัวตนที่ปลอมยาก.** การผูกสิทธิ์เข้ากับ key/certificate บน
   ฮาร์ดแวร์แท้ ทำให้ปลอม "ตัวตนที่เชื่อถือได้" ได้ยาก

---

## 9. การใช้งานที่ถูกต้องตามกฎ (Legitimate Use)

ถ้าเป้าหมายคือ "อยากได้ทีมไว้เล่น/ทดสอบ" โดยไม่ยุ่งกับบริการออนไลน์:

- **Pokémon Showdown** — เล่นแข่งออนไลน์ฟรี ใช้ Showdown set ได้ตรงๆ ไม่ต้องมีเครื่อง
  และไม่แตะ HOME เลย
- **Emulator / เครื่องของตัวเอง** — ทดสอบทีมในโหมด single-player

เครื่องมือในรีโพนี้รองรับเคสเหล่านี้ได้โดยไม่ต้องเกี่ยวข้องกับ Layer 3 เลย

---

## 10. สรุป

กรณีศึกษานี้แสดงให้เห็นว่า **"ควบคุม client ได้ 100% ≠ ทะลุระบบได้"** ตราบใดที่:

1. source of truth อยู่ฝั่ง server
2. server ตรวจสอบข้อมูลขาเข้าเอง
3. ช่องทางเข้าออกถูก gate ด้วยการพิสูจน์ตัวตนที่ปลอมยาก

การ "แก้เซฟได้แต่ยัดเข้า HOME ไม่ได้" จึงไม่ใช่ช่องโหว่ที่ยังหาไม่เจอ แต่เป็น
**ผลลัพธ์ที่ถูกต้องของการออกแบบ trust boundary ที่ดี** — ซึ่งเป็นบทเรียนที่นำไปใช้
ออกแบบระบบ client–server ปลอดภัยได้ทั่วไป
